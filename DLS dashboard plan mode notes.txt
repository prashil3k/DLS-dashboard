# DLS Performance Dashboard — Rebuild Plan

## Context

Storylane has ~9,000+ tutorial pages driving SEO traffic that's been declining for months. A prototype dashboard exists (`DLS performance dashboard/`) with 4 Streamlit tabs, but all data is hardcoded in `sample_data.py`. The goal is to make this a live, data-driven diagnostic tool that answers: which clusters are dying, why (AIO vs brand displacement vs staleness), and where to invest.

**Key discovery**: The Ahrefs GSC MCP integration is free (0 units per row), has 16+ months of history, and gives us both page-level and keyword-level data with article URL mapping. This replaces the need for direct GSC OAuth setup entirely.

**Region**: United States (baseline for all queries)
**"Lost to" analysis**: Pinned for later — build the core dashboard first
**HubSpot**: Skipped for now

---

## Data source: Ahrefs GSC MCP tools

Confirmed working with real data. Project ID: `9000301` (Storylane).

| Tool | What it gives us | Use case |
|------|-----------------|----------|
| `gsc-pages` | Per-article: URL, keyword count, top keyword, clicks, impressions, CTR, position, traffic value | Monthly article snapshots → cluster aggregation |
| `gsc-keywords` | Per-keyword: keyword, top_url, clicks, impressions, CTR, position | Keyword-level drill-down per cluster |
| `gsc-keyword-history` | Per-keyword time-series: monthly clicks, impressions, CTR, position | Trend lines for specific keywords |
| `gsc-page-history` | Per-page time-series | Article-level trends |
| `gsc-performance-history` | Project-wide time-series | Overall health check |

All tools: 0 API units, country filter supported, date ranges from Dec 2024+.

## Phase 1: Data pipeline — backfill + SQLite

### 1a. Backfill via MCP (orchestrated by Claude Code)

For each month Jan 2025 → May 2026 (17 months):
1. Call `gsc-pages` with `project_id=9000301, country=US, date_from=YYYY-MM-01, date_to=YYYY-MM-last, limit=5000, where={"field":"url","is":["isubstring","/tutorials/"]}`
2. Save response as `data/pages/pages_YYYY-MM.json`
3. Call `gsc-keywords` with same params + `limit=1000`
4. Save as `data/keywords/keywords_YYYY-MM.json`

This is ~34 MCP calls total (17 months × 2 tools). All free.

### 1b. SQLite schema

**`pages_monthly`** — one row per page per month:
```sql
CREATE TABLE pages_monthly (
    page TEXT NOT NULL,
    month TEXT NOT NULL,           -- YYYY-MM
    cluster TEXT,
    keywords_count INTEGER,
    top_keyword TEXT,
    clicks INTEGER DEFAULT 0,
    impressions INTEGER DEFAULT 0,
    ctr REAL DEFAULT 0,
    position REAL DEFAULT 0,
    traffic_value REAL,
    PRIMARY KEY (page, month)
);
CREATE INDEX idx_pm_cluster_month ON pages_monthly(cluster, month);
```

**`keywords_monthly`** — one row per keyword per month:
```sql
CREATE TABLE keywords_monthly (
    keyword TEXT NOT NULL,
    month TEXT NOT NULL,
    top_url TEXT,
    cluster TEXT,
    urls_count INTEGER,
    clicks INTEGER DEFAULT 0,
    impressions INTEGER DEFAULT 0,
    ctr REAL DEFAULT 0,
    position REAL DEFAULT 0,
    PRIMARY KEY (keyword, month)
);
CREATE INDEX idx_km_cluster_month ON keywords_monthly(cluster, month);
```

**`cluster_config`** — 17+ clusters with URL matching rules:
```sql
CREATE TABLE cluster_config (
    cluster TEXT PRIMARY KEY,
    url_patterns TEXT NOT NULL,     -- JSON array of slug substrings
    parent_domain TEXT              -- e.g. 'canva.com' for later "lost to" analysis
);
```

### 1c. Clustering logic (`cluster.py`)

Rule-based: extract tool name from URL slug (`/tutorials/how-to-X-in-[tool]`), fallback to query text.


### 1d. Ingestion script (`ingest.py`)

Reads all JSON files from `data/pages/` and `data/keywords/`, assigns clusters, upserts into SQLite. Idempotent — safe to re-run.

### 1e. Database query layer (`db.py`)

Key functions:
- `cluster_summary(periods: dict) → DataFrame` — multi-period comparison table
- `cluster_monthly_trends(cluster) → DataFrame` — monthly time-series for a cluster
- `keyword_movers(cluster, period_a, period_b) → DataFrame` — biggest position changes
- `position_distribution(cluster, month) → dict` — count of queries in top 1.5/3/5/10

Periods for the overview table:
- **Post-peak Q1'25**: 2025-02-01 to 2025-04-30
- **Q1'26**: 2026-01-01 to 2026-03-31
- **3 months ago**: rolling window (current_month - 4) to (current_month - 2)
- **Current**: last full month of data in DB

## Phase 2: Dashboard views

### Tab 1 — Cluster overview (multi-period comparison)

Table with all clusters, each showing 4 time periods side by side:
| Cluster | Clicks Q1'25 | Clicks Q1'26 | Clicks 3mo ago | Clicks current | Delta | Impressions... | CTR... | Avg Pos... |

Color-coded: green (improving), red (declining), yellow (flat). Sortable by any column. Click a cluster → jumps to Tab 2 for that cluster.

Also: bubble chart (impressions vs CTR, sized by clicks) and horizontal bar chart (clicks by cluster).

### Tab 2 — Ranking tracker (replaces Position Health)

For a selected cluster:
- **Position bucket chart**: Line chart, x=month, lines = count of keywords in top 1.5 / top 3 / top 5 / top 10. Shows Canva's collapse (299→18 at top 1.5) visually.
- **Avg position trend**: Monthly average position for the cluster
- **Position movers table**: Keywords sorted by `impressions × abs(position_change)` between two selected periods. Volume-weighted so a rank 1→2 shift on 50K impressions surfaces above noise.

### Tab 3 — AIO signature detector

For a selected cluster, three lines on one chart: clicks, CTR, avg position over time.
- Position holds + CTR drops → AIO flag (red alert)
- Position drops + CTR drops → brand displacement (yellow)
- Both stable → healthy (green)

Works for all clusters now (data from SQLite, not hardcoded for just Notion/Canva).

### Tab 4 — Query deep dive

For a selected cluster + time period:
- Scatter: impressions (x) vs CTR (y), sized by clicks, colored by signal (healthy/AIO/displaced)
- Table: all keywords with clicks, impressions, CTR, position, signal classification
- Signal auto-classification from GSC data: position ≤ 3 + CTR < 1% = likely AIO; position dropped 3+ = displaced; else = healthy
- Toggle between clusters via dropdown

## Phase 3: "Lost to" analysis (pinned for later)

Three buckets — requires SERP data beyond GSC:
1. **Lost to AIO**: Position stable, CTR dropped >30% → detectable from GSC alone
2. **Lost to parent domain + YouTube**: Ahrefs `serp-overview` for top queries per cluster
3. **Lost to unbranded + reddit + forums**: Everything else

Will revisit when credit budget is clear. Bucket 1 can be built immediately as part of Tab 3/4.
